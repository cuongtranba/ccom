# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Tests
bun test                                    # full suite (334 tests, 19 files)
bun test packages/node                      # node tests only
bun test packages/server                    # server tests (requires Redis)
bun test packages/node/test/engine.test.ts  # single file

# Dev
bun run server          # start central WebSocket server
bun run node serve      # start node MCP channel server
bun run init            # interactive config wizard
docker-compose up -d    # Redis + server

# Build & Publish
bun run build:node      # bundles packages/node + shared → dist/cli.js
bun run publish:node    # publish @tini-works/inv-node to GitHub Packages
```

## Architecture

**Monorepo** with 4 packages: `shared` (types + protocol), `server` (WebSocket hub), `node` (local client + MCP), `dashboard` (Astro SSR UI).

### Message Flow

```
Claude → MCP stdio → channel.ts → Engine → Store (SQLite)
                                      ↓
                                   WSClient → Central Server (Redis-backed) → Other Nodes
```

- `channel.ts` — MCP server exposing 19 tools via `@modelcontextprotocol/sdk` stdio transport
- `engine.ts` — orchestrates store + state machines + signal propagation (40+ methods)
- `store.ts` — SQLite CRUD via `bun:sqlite` prepared statements (13 tables, 50+ methods)
- `ws-client.ts` — WebSocket with exponential backoff reconnect
- `ws-handlers.ts` — dispatches inbound envelopes from server to engine

### State Machines

**Items**: `unverified → proven → suspect → broke` (5 transitions in `state.ts:StateMachine`)

**Change Requests**: `draft → proposed → voting → approved/rejected → applied → archived` (7 transitions in `state.ts:CRStateMachine`)

### Signal Propagation

When an item changes state, `SignalPropagator` recursively marks all downstream traced items as `suspect` (breadth-first with cycle prevention via visited set).

### Server Routing

Central server (`packages/server`) routes envelopes:
1. Target node online on this instance → direct `ws.send()`
2. Online on another instance → Redis pub/sub
3. Offline → Redis outbox queue, drained on reconnect

### Vertical Hierarchy

`pm → design → dev → qa → devops` — defined in `UPSTREAM_VERTICALS`. Traces flow downstream; signals propagate through this dependency chain.

## Key Conventions

- **Runtime**: Bun only — uses `bun:sqlite`, `bun:test`, native WebSocket
- **Database**: Raw SQL with `bun:sqlite` prepared statements, no ORM. Row types use `snake_case`, domain types use `camelCase` with mapper functions.
- **IDs**: UUID v4 via `crypto.randomUUID()`
- **Timestamps**: ISO strings via `datetime('now')`
- **Tests**: In-memory SQLite (`:memory:`) for unit tests. `createTestNode()` + `registerRemoteItem()` helpers for multi-node integration tests.
- **Protocol**: 23 message payload types in `shared/src/messages.ts`, wrapped in `Envelope` (messageId, fromNode, toNode, projectId, timestamp, payload)
- **Config**: `inv-config.json` for node identity + server URL + autonomy rules. `.mcp.json` auto-generated by wizard for Claude auto-discovery.

## Adding a New MCP Tool

1. Add schema to `TOOL_DEFINITIONS` array in `channel.ts`
2. Add case to `handleCall()` switch in `channel.ts`
3. Implement logic in `engine.ts`
4. If it handles network messages, add handler in `ws-handlers.ts`
5. Test in `channel.test.ts`

## Package: @tini-works/inv-node

Published to GitHub Packages. Build bundles `@inv/shared` inline, keeps `@modelcontextprotocol/sdk` external. CLI subcommands: `init` (wizard), `serve` (MCP server), `update` (clear cache). Auto-released on push to main via `.github/workflows/release-node.yml`.
